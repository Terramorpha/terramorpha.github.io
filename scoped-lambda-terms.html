Well scoped lambda terms

In (Scott, Phil and Obua, Steven and Fleuriot, Jacques, 2017), I stumbled upon the following type declarations1: 

data Var a = Z | S a
  deriving (Show, Functor)

data Term t a = V a
  | Abs (Term t (Var a))
  | Fix (Term t (Var a))
  | App (Term t a) (Term t a)
  | Prim (t (Term t a))
  deriving (Functor)

This made me finally understand the "Syntax is an endofunctor on the category of substitutions" point of view. In this post, I will try to condense as much as possible the parts I would have needed to understand it sooner.

1. The anatomy of the term

This datatype defines the syntax of well-scoped untyped λ-calculus terms where the free variables live in an alphabet \( a \) and primitive values come from a \( t \) functor.

The weird constructor was interesting to me, because it is strangely similar to how the free monad is usually defined: 

data Free f x = Pure x
  | In (f (Free f x))

This is probably not a coincidence, because the goal of this constructor is to let a user extend the language with new (possibly recursive) terms. This is similar to one usecase of free monads: building syntax trees to be evaluated later.

2. Renaming and substitution

One interesting fact about this presentation of λ-terms is the fact that the "renaming" operation that one will often see when dealing with λ-calculus is simply the functorial action of on free variables: 

rename :: Functor t => (a -> b) -> Term t a -> Term t b
rename = fmap

At first glance(at least for me), this looks very incorrect. How could a simple fmap deal with the intricacies of correct renaming of (for instance) Abs terms?

Let's say one has a(constant) renaming function \( r : \left\{ a \right\} \to \left\{ b \right\} \). Applying \( \text{fmap}(r) : \text{Term}(t, \left\{ a \right\} ) \to \text{Term}(t, \left\{ b \right\} ) \) on \( \lambda.\#0 \) would yield… \( \lambda.\#0 \). This is exactly the correct behaviour. This works because is itself a functor(The deriving form reuses the Functor Var instance).

Now that we know how to do renaming, let's do substitution. 

sub :: Functor t => (v1 -> Term t v2) -> Term t v1 -> Term t v2

(The instance is necessary to consider itself a functor)

The first few cases are easy to do 

sub s (V ref) = s ref
sub s (App a b) = App (sub s a) (sub s b)
sub s (Prim b) = Prim (fmap (sub s) b)

The harder ones are where we need to manage bound variables in and terms. We need the usual function. 

sub s (Abs body) = Abs (sub (shiftSub s) body)
sub s (Fix body) = Fix (sub (shiftSub s) body)

This one lets us "lift" our substitution to a context with one more free variable. 

shiftSub :: Functor t => (a -> Term t b) -> Var a -> Term t (Var b)
shiftSub f Z = V Z
shiftSub f (S r) = rename S (f r)

If we look back at the signature of our function. We notice it has (apart from the order of the arguments) the same shape as the operator. Indeed, we have a monad!

If is our bind operator, we still need the . 

termRet :: a -> Term t a
termRet = V

3. An example

-- Example

data Null
  deriving Show

data Empty a
  deriving (Show, Functor)

body :: Term Empty (Var Null)
body = (App (V Z) (V Z))

func :: Term Empty Null
func = Abs body

x = sub (const func) body

4. Show

If you want to experiment with this code, a Show instance will be useful 

instance (forall x. Show x => Show (t x), Show a) => Show (Term t a) where
  show (V a) = show a
  show (Abs b) = "\\." ++ show b
  show (Fix b) = "\\m." ++ show b
  show (App a b) = "("++show a++","++show b++")"
  show (Prim b) = show b
