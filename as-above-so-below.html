As above, so below: An exploration of the anatomy of a preemptive, asynchronous multitasking system

Most high level programming languages use some kind of run time library. This library is responsible for serving as a bridge between the programming language’s ontological universe and the operating system’s and will. Such a run time library often provides a set of mutually interdependent functionality.

A program is a sequence of instructions that either do some computations (normal instructions) or invoke system calls, letting them write to disk, to the network, to the screen et cetera. Some input/output operations, however, take a long time to finish and, if when performed the normal way, will just completely block the execution of the process until they complete. Some run time systems try to use this “wasted time” by doing other work while the progam waits for I/O to complete. However, without expanding the “sequence of instructions” computing model, it is unclear what that “other work” might be. Many programming language let the user express many independent pieces of computation through some task-like abstraction. When many tasks have work that need to be done, one can make progress on a second task while the first is blocked by I/O.

A run time system, because it is reproducing functionality that is similar to the operating system’s, has very similar organs and by studying (through implementation) a run time system, one can better understand the design of operating systems. 

const std = @import("std");
const testing = std.testing;
const linux = std.os.linux;

1. I/O

The entrance of the rabbit hole we are going to explore is input and output. The first thing we want to do is pretty simple, we want to separate the action of starting an I/O operation from the part where we wait for it to complete. In code, that means somehow turning this: 

var buffer: [16]u8 = undefined;

const n = try std.fs.File.stdin().read(buffer[0..]);

into that: 

var buffer: [16]u8 = undefined;

var readOperation = std.fs.File.stdin().readOperation(buffer[0..]);

// Do some stuff

const n = readOperation.wait();

The API in the second block of code is imaginary, but you get the idea. Why would we want to separate the registration of an operation from waiting for its completion, though? Because instead of waiting (suspending the process/thread) for the operation to complete, you might be interested in doing something more useful. The Linux kernel provides multiple syscalls/systems to do this kind of separation.

1.1. fcntl and O_NONBLOCK

We have a syscall fcntl which lets us modify the configuration associated to a file descriptor. In particular, it lets us turn on the O_NONBLOCK option which makes every read that would block instead return EAGAIN. You would then have a loop that goes through a bunch of those file descriptors and busy waits until they are all empty. However, this pattern is inefficient and moreover, it only works for networks sockets (and other kinds of file descriptors which have a obvious “wait for stuff to arrive” phases) and will always block on file reads even when the hard disk is slow.

1.2. poll / epoll

If you have a bunch of file descriptors in an array with corresponding operations, you might want to iterate through them and try (in a nonblocking way) to perform the operations with the previous technique. However, a better way would be to use epoll and tell the kernel something like “here are a bunch of file descriptors I’m interested in, wait until something happens to any of them”. poll operates on an array of file descriptors + metadata () while epoll uses a more efficient data structure, but they are essentially the same. you can think of those syscalls as a more efficient implementation of looping on file descriptors with O_NONBLOCK, so they inherit the problems of this method, amongst other things, they won’t let you read asynchronously from files.

1.3. aio_*

The aio_* family of syscalls, like the name suggests, actually let you do asynchronous I/O instead of just telling you if some kernel buffer associated to a socket contains stuff. The api contains those syscalls:

  • aio_read(), aio_write() - Start async I/O
  • aio_suspend() - Wait for completion
  • aio_error() - Check if operation succeeded
  • aio_return() - Get result (bytes transferred)
  • aio_cancel() - Cancel pending operation

and they work similar to the imaginary api I motivated this section with. aio_suspend even lets you pass multiple events to wait for, at the same time.

1.4. io_uring

Modern systems, however, prefer to use the modern linux api io_uring and it is this one which we will focus on in this work. the io_uring system, however is very complicated. Here is a brief overview of how to work with it:

  1. We use io_uring_setup to create a io_uring instance in the kernel and get a handle to it.
  2. We mmap three different regions of memory within the handle. These regions are shared with the kernel and contain three things:
    1. A shared buffer of submission queue entries (io_uring_sqe) which we will use to store all the metadata related to operations.
    2. A shared ring buffer – the submission queue – containing indices into the buffer of submission queue entries.
    3. A shared ring buffer – the completion queue – containing completion queue entries (io_uring_cqe) which hold the information of completed operations.
  3. When we want to submit an operation, we choose some slot in the submission queue entries buffer, write the operation type, file descriptor and associated buffer, then we write the index of this entry into the submission queue and finally we call io_uring_submit.
  4. Operations that complete will be written asynchronously by the kernel into the completion queue. If we want to wait for a number of operations to complete, we can use io_uring_enter.

While this whole setup is a lot more complicated than the previous ones, this complexity gets us big efficiency gains. Because we share a slice of memory with the kernel and because that specific type of queue can be used without synchronization, it is possible for the process-side to poll for completions without performing syscalls by just reading the completion queue and checking if the length is nonzero.

Because the io_uring family is the most modern and efficient, it is this one we are going to use for our asynchronous I/O.

2. Task switching

Writing programs that work directly with the raw async IO primitives is not fun and we want to make it more ergonomic to express asynchronous IO. One way of doing that is to introduce the concept of tasks, which are independent subprograms that perform IO independently. Our system will execute a task until it tries to perform an operation and instead of blocking, it will register the operation and give control to an other task. At some point, the other task will try to perform IO, the first operation will have succeeded and the system will cede control again to the first task. We currently know how to use io_uring to do IO asynchronously, but we still need to properly define what we mean by a “task” and how we “cede control” to an other task.

Let’s say we have a way to save and load a program state such that this state contains everything important about that program, then we would be able to perform a task switch by saving the current state and loading an other state. More concretely, if we save the current task’s (task ) context somewhere (let’s say, at global variable ctx_a, then load in task ’s context (global variable ctx_b), we can execute task for some time, then go back to executing task by loading the context from ctx_a. In other words, here is what we want our task switching API will look like: 

const Task = struct {
    // ...

    fn switchTo(me: *@This(), new: *const @This()) void {
        // ...
    }
};

To a first approximation, the state of a program is just the values of all the registers of the CPU. In theory, if we saved the state of all the registers it would be sufficient to resume execution later in the correct state. However, when we only wish to switch tasks at certain points in the program – let’s say, at a function call boundary – we don’t actually need to save all of them. The system V calling convention divides all the cpu registers into two categories, those that should be saved by the caller and those that should be conserved by the callee. This categorization makes it so that the assembly associated to a function that has just been called knows exactly which registers it can use for computation (registers it can clobber the values of as much as it wants) and which registers it must keep the values of.

Those callee-saved registers are:

  • rsp is the stack pointer. When the function returns, the stack pointer must have the same value as just before the function was called.
  • rbp is the base pointer. If the callee modifies this register (to use its own stack frame), it will need to remember the old value.

The rest are general purpose

  • rbx
  • r12
  • r13
  • r14
  • r15

Okay, so we know that what we need to save is exactly the callee-saved registers under the system V abi, but how do we save those registers, exactly? 

const Context = extern struct {
    rsp: usize,
    rbp: usize,
    rbx: usize,
    r12: usize,
    r13: usize,
    r14: usize,
    r15: usize,
};

It is rare in programming that we actually need to operate at the level of registers (even when doing low-level systems programming), but it is the case here. Most low level languages let you write assembly inside your program and Zig does too.

Assuming that:

  • The %rdi register is the address where we want to save the current context.
  • The %rsi register is the address from which we want to load the context.

the following piece of assembly will save the current context, then immediately load the next one. 

fn swapContextAsm() callconv(.naked) void {
    asm volatile (
    // Save current context (old is in %rdi, new is in %rsi per C ABI)
        \\movq %%rsp, %[rsp_offset:c](%%rdi)
        \\movq %%rbp, %[rbp_offset:c](%%rdi)
        \\movq %%rbx, %[rbx_offset:c](%%rdi)
        \\movq %%r12, %[r12_offset:c](%%rdi)
        \\movq %%r13, %[r13_offset:c](%%rdi)
        \\movq %%r14, %[r14_offset:c](%%rdi)
        \\movq %%r15, %[r15_offset:c](%%rdi)
        // Load new context
        \\movq %[rsp_offset:c](%%rsi), %%rsp
        \\movq %[rbp_offset:c](%%rsi), %%rbp
        \\movq %[rbx_offset:c](%%rsi), %%rbx
        \\movq %[r12_offset:c](%%rsi), %%r12
        \\movq %[r13_offset:c](%%rsi), %%r13
        \\movq %[r14_offset:c](%%rsi), %%r14
        \\movq %[r15_offset:c](%%rsi), %%r15
        // return
        \\  retq
        :
        : [rsp_offset] "i" (@offsetOf(Context, "rsp")),
          [rbp_offset] "i" (@offsetOf(Context, "rbp")),
          [rbx_offset] "i" (@offsetOf(Context, "rbx")),
          [r12_offset] "i" (@offsetOf(Context, "r12")),
          [r13_offset] "i" (@offsetOf(Context, "r13")),
          [r14_offset] "i" (@offsetOf(Context, "r14")),
          [r15_offset] "i" (@offsetOf(Context, "r15")),
        : .{ .memory = true });
}

Now, the astute reader might say: “but when we load the context, how will the code know what instruction pointer to use”? If we arrange for every context to be captured exclusively after a callq instruction, then the stack pointer of a context will point (the first item on the stack) to the return address of the call instruction. We specify the calling convention to be .naked specifically so that the compiler doesn’t generate a preamble that would mess up a bunch of registers and break our “return address is on top of the stack” invariant. Having a .naked convention also forces us to call this piece of code from assembly which motivates the following wrapper function: 

fn swapContext(old: *Context, new: *const Context) void {
    // After loading the address
    asm volatile (
        \\ callq %[asmCode]
        :
        : [_] "{rdi}" (old),
          [_] "{rsi}" (new),
          [asmCode] "r" (&swapContextAsm),
        : .{ .memory = true });
}

The problem we have, however, is that even if our zsh:1: command not found: zig primitive works, we still need a second context to use it with. This time, we will be creating a context from scratch, using a given slice of memory as the stack and taking some given function as an entrypoint. The challenge here, will be setting up everything correctly.

The main constraint is that the newly created context should have an instruction pointer which resumes execution as its first item. One idea would be to stick the address of the function we want to call in the top slot of the stack, but it won’t work, because we want to pass an argument to the function and our context doesn’t capture the argument holding registers. The way we are going to solve this is through a “trampoline” which will receive all the data on the stack, will set the correct registers from this data and finally will jump to the function. 

fn initContext(
    stack: []u8,
    func: *const fn (usize) callconv(.c) noreturn,
    data: usize,
) Context {

    // The stack grows downwards, so the pointer must point to the end
    // address (exclusive) of the valid range. It must also be
    // word-aligned, so we mask out the last four bits of the end
    // address.

    const start_addr: usize = @intFromPtr(stack.ptr);
    const end_addr: usize = start_addr + stack.len;
    const aligned_end_addr = end_addr & ~@as(usize, 0b1111);
    var stack_pointer: [*]usize = @ptrFromInt(aligned_end_addr);

    // Because the stack grows downwards, %rsp always points to the
    // beginning address of the top element. To push something on the
    // stack, we must therefore decrement the pointer by the size of
    // the thing we want to push, then write the thing at that new
    // address.

    // On the bottom of the stack, we write the address of the code we
    // want to be executed.
    stack_pointer -= 1;
    stack_pointer[0] = @intFromPtr(func);

    // In the middle, we write the userdata (a single machine word).
    stack_pointer -= 1;
    stack_pointer[0] = data;

    // On top of the stack, we write the address of the trampoline.
    // This element will be popped by the `ret` instruction as soon as
    // the context is loaded for the first time.
    stack_pointer -= 1;
    stack_pointer[0] = @intFromPtr(&initContextTrampoline);

    var ctx = std.mem.zeroes(Context);

    // The only nonzero register here is the stack pointer, the rest
    // will be set up by the trampoline.
    ctx.rsp = @intFromPtr(stack_pointer);

    return ctx;
}

This code sets up a stack such that we have the following layout (remember: the “top” of the stack is the bottom most address):

func
data
trampoline 
fn initContextTrampoline() callconv(.naked) void {
    asm volatile (
        \\ pop %%rdi
        \\ pop %%rsi
        \\ callq %%rsi
        ::: .{});
}

Here, we load data into %rdi, func into %rsi and we jump to the address at %rsi. An important detail to notice here is the fact that we required func to use the .c calling convention. This calling convention demands that the first argument of a function be passed through the %rdi register, and it is this guarantee that makes it possible to properly pass an argument to func from the inline assembly.

Now, let’s play with our Context primitive. The workflow is going to look like this:

  1. Allocate the memory for the stack. It can be in the heap, or on the stack itself, but it should be big enough so that you don’t overflow it.
  2. Initialize a “child context” with that stack memory. Pass a function pointer together with a pointer to everything that function needs to work.
  3. swapContext to that child context and save the parent context somewhere.
  4. When the child has executed, it should give the control back to an other context. The noreturn enforces that the function never returns, because if it did (say, if the return type was void), it would try to read a return address just after the end of its allocated stack, would read garbage, would try to jump to garbage and would segault. This would be bad, so instead, we use the type system to enforce that the function should take care of itself.
test "Context" {
    const C = struct {
        var parent_context: Context = undefined;
        var child_context: Context = undefined;
        var flag: bool = false;

        fn func(_: usize) callconv(.c) noreturn {
            flag = true;
            swapContext(&child_context, &parent_context);
            unreachable;
        }
    };

    var stack: [1 << 20]u8 = undefined;

    C.child_context = initContext(stack[0..], C.func, 0);

    swapContext(&C.parent_context, &C.child_context);

    try testing.expect(C.flag);
}

3. Asynchronous IO ❤️ task switching

Now that we have primitives that let you create multiple program contexts and switch between them, let’s use it in tandem with the io_uring system to truly make asynchronous IO seamless.

3.1. Zig’s IoUring implementation

The Zig standard library contains code that manages the io_uring memory maps for us, so we can quickly get to use it. Here is what the workflow of using IoUring looks like: 

test "IoUring" {
    var uring: linux.IoUring = try .init(1 << 5, 0);
    defer uring.deinit();

    var fds: [2]linux.fd_t = undefined;

    _ = linux.pipe(&fds);

    const can_read = fds[0];
    const can_write = fds[1];

    var buffer: [16]u8 = undefined;

    // We queue a bunch of operations
    _ = try uring.read(12345, can_read, .{ .buffer = buffer[0..] }, 0);

    // We submit the operations in a single call.
    _ = try uring.submit();

    // The other side of the socket writes
    const test_string = "this is a test";

    _ = linux.write(can_write, test_string, test_string.len);

    // We wait for something to happen
    _ = try uring.enter(0, 1, 0);

    // For all the completed operations, we look at the results...
    const cqe = try uring.copy_cqe();

    // The user data attached to the operation should be the right one
    try testing.expectEqual(12345, cqe.user_data);

    // The read operation should have completed (with the right )
    try testing.expectEqualStrings(test_string, buffer[0..@intCast(cqe.res)]);
}
